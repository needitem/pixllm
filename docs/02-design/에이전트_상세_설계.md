# 에이전트 상세 설계

기준일: 2026-04-06

이 문서는 현재 PIXLLM의 agent 구조를 설명한다.

## 1. 현재 에이전트 정의

현재 desktop chat 경로에는 하나의 주 agent loop만 있다.

- `QueryEngine`: 모델 호출과 turn loop
- `ToolRuntime`: tool authorization, grounding, batch execution

즉 현재 PIXLLM은 여러 개의 독립 LLM worker가 협업하는 구조가 아니라, `하나의 local coding agent + Qwen adapter + 도구 집합 + backend evidence plane` 구조다.

## 2. 구성 요소

| 구성 요소 | 역할 |
|---|---|
| `QueryEngine.cjs` | turn loop, transcript, retries, compaction, continuation, grounding retry |
| `query.cjs` | block helper와 adapter facade |
| `QwenAdapter.cjs` | textual tool-call parsing, limited recovery, transcript flattening |
| `processUserInput.cjs` | request context 생성 |
| `ToolRuntime.cjs` | active tool scope, central permission, tool batch 실행, synthetic recovery |
| `StreamingToolExecutor.cjs` | streaming 중 parse 가능한 tool call 선실행 |
| `nextSpeakerCheck.cjs` | prose-only 응답 뒤 continuation 판단 |
| `useCanUseTool.cjs` | 중앙 deny-by-default gate |
| `tools/*` | 실제 file/shell/build/reference/task tool |
| `taskRuntime.cjs` | background task 상태 관리 |
| `queryTrace.cjs` | grounded evidence와 trace 요약 |

## 3. 실행 생명주기

1. 사용자가 요청을 보낸다.
2. `queryEngineService`가 세션별 `QueryEngine`을 준비한다.
3. `processUserInput`가 request context를 만든다.
4. `processUserInput`가 필요할 때 `languageProfile`과 `symbolHints`를 만든다.
5. `QueryEngine`이 시스템 프롬프트, transcript, project context를 조합해 모델을 호출한다.
6. `QwenAdapter`가 응답을 text, tool call block, reasoning-based recovery candidate로 파싱한다.
7. `StreamingToolExecutor`가 parse 가능한 일부 tool call은 미리 실행한다.
8. `ToolRuntime`이 permission을 다시 확인하고 tool result를 transcript에 기록한다.
9. prose-only 응답이면 `nextSpeakerCheck`가 model/user 중 누가 다음 화자인지 판정한다.
10. grounded answer가 가능하면 최종 응답을 반환한다.

## 4. request context

현재 request context에는 아래 정보가 들어간다.

- prompt
- workspace path
- selected file path
- directives
- explicit path candidates
- allowed direct paths
- evidence preference
- intent flags
- `languageProfile`
- `symbolHints`
- initial/active tool names
- user-facing summary

이 정보는 system prompt, active tool scope, permission gate, parser recovery에 같이 쓰인다.

## 5. permission model

현재 agent는 중앙 gate와 tool hook을 같이 쓴다.

- 중앙 gate는 unknown path, backend-only path, execution intent, turn별 active tools를 검사한다.
- 개별 tool은 schema validation, output schema, stale-read, user-facing summary를 제공한다.
- 기본 정책은 fail-closed에 가깝다. 새 tool이 명시 policy 없이 추가되면 막힌다.

## 6. Qwen 적응층의 위치

현재 Qwen 적응층은 agent loop 바깥의 임시 패치가 아니라, agent의 일부다.

- `QwenAdapter`는 tool intent를 block transcript로 변환한다.
- `processUserInput`는 한국어 질문을 별도 rewrite helper 없이 language-aware request context로 정리한다.
- `nextSpeakerCheck`는 prose-only turn 뒤에 continuation 여부를 재판정한다.

즉 현재 PIXLLM의 agent는 `model-neutral agent`가 아니라 `Qwen-first coding agent`로 이해하는 것이 정확하다.

## 7. backend evidence와 agent의 관계

`company_reference_search`는 backend의 `lookup_sources_and_code`를 호출한다.

- 회사 엔진 소스나 내부 문서를 backend에서 검색한다.
- 결과는 desktop transcript에 read-only evidence로 편입된다.
- backend path는 grounded source로는 인정되지만 곧바로 local edit 대상이 되지는 않는다.

따라서 backend는 두 번째 agent가 아니라 evidence provider다.

## 8. task 개념

현재 `task_*`는 별도 LLM agent가 아니다.

- long-running shell/build command 추적
- 상태 변경
- terminal output 확인

즉 `Task Agent`, `Verifier Agent`, `Remote Agent`, `Team Worker` 같은 개념은 현재 기본 채팅 경로에 없다.

## 9. compaction과 생존성

Qwen server는 flattened messages 안에 user query가 사라지면 `No user query found in messages` 오류를 낼 수 있다. 현재 desktop agent는 다음을 보장한다.

- message compaction 시 첫 substantive user message 보존
- flattened payload에 user query가 없으면 fallback `User request:` 메시지 주입

이 장치는 긴 세션과 transcript compaction이 겹칠 때 agent loop를 살려주는 현재 구현의 중요한 안정화 포인트다.

## 10. 현재 없는 에이전트 계층

현재 코드 기준으로 구현되지 않은 것:

- multi-agent orchestration
- remote agent / bridge session
- MCP agent registry
- dedicated verifier worker
- backend answer-generation agent

현재 PIXLLM의 정확한 설명은 `단일 desktop Qwen coding agent`다.
