# 에이전트 상세 설계

기준일: 2026-04-01

이 문서는 현재 구현된 PIXLLM 에이전트 구조를 설명합니다.

## 1. 현재 에이전트 정의

현재 desktop local chat 경로에는 사실상 하나의 주 에이전트만 있습니다.

- `LocalAgentEngine`: 한 턴의 모델 호출과 반복 제어
- `LocalAgentRuntime`: request context, tool policy, grounding, tool batch 실행

즉 현재 에이전트는 여러 LLM worker의 집합이 아니라, `엔진 + 런타임 + 로컬 도구 registry` 조합입니다.

## 2. 현재 구성 요소

| 구성 요소 | 역할 |
|---|---|
| `LocalAgentEngine` | 모델 호출, assistant 파싱, turn budget, retry, finalization |
| `LocalAgentRuntime` | pre-loop context, permission, grounding, tool batch orchestration |
| `LocalToolCollection` | 도구 정의/검색/실행 |
| `local_task_runtime` | background shell/build task 추적 |
| `workspace.cjs` | 파일/검색/셸/빌드 작업의 실제 실행 |

## 3. 현재 실행 수명주기

1. 사용자가 데스크톱 UI에서 요청을 보냅니다.
2. `LocalAgentService`가 엔진 실행을 시작합니다.
3. 런타임이 request context를 만듭니다.
4. 엔진이 모델을 호출합니다.
5. assistant 응답이 `text`와 `tool_use`로 파싱됩니다.
6. 런타임이 tool permission을 검사하고 도구 batch를 실행합니다.
7. trace와 transcript가 갱신됩니다.
8. grounded answer가 가능해지면 최종 응답을 반환합니다.

## 4. 현재 request context

현재 런타임이 관리하는 request context:

- prompt
- workspace path
- selected file path
- explicit path references
- allowed direct paths
- intent flags
- UI/runtime context summary

이 컨텍스트는 모델 메시지와 tool policy 양쪽에 동시에 쓰입니다.

## 5. 현재 tool permission 모델

현재 에이전트는 모든 tool call 전에 중앙 policy를 거칩니다.

주요 규칙:

- 사용자 요청이나 prior evidence에 없는 경로는 바로 읽지 않음
- 파일은 먼저 읽고 나서 편집
- shell/build는 execution intent 또는 충분한 context가 있을 때만 허용
- grounded path 집합 밖의 경로를 답변에 바로 언급하지 않음

## 6. 현재 task 개념

현재 `task_*`와 background shell/build는 존재하지만, 이는 별도 LLM sub-agent가 아닙니다.

- long-running command 추적
- terminal output 보존
- 상태 조회와 중지

즉, 현재 `Task Agent`, `Verifier Agent`, `Remote Agent`, `Team Worker` 같은 분리된 에이전트 계층은 로컬 채팅 경로에 구현되어 있지 않습니다.

## 7. 현재 구현되지 않은 에이전트 계층

문서나 설계 초안에 남아 있던 아래 개념은 현재 코드에 없습니다.

- team worker 기반 병렬 LLM 실행
- remote agent / bridge session
- verifier 전용 별도 agent
- plugin/skill/MCP agent registry

이들은 장기 설계 후보일 수는 있지만, 현재 PIXLLM을 설명할 때는 기본 개념으로 두면 안 됩니다.

## 8. 현재 상태 요약

현재 PIXLLM 에이전트는 `단일 로컬 코딩 에이전트`입니다.

- 강점: request context, tool policy, grounded answer, structured trace
- 약점: streaming tool execution 부재, backend runtime과 분리, multi-agent 부재

따라서 현재 문서 기준의 정확한 표현은 `멀티 에이전트 플랫폼`보다 `강화된 단일 로컬 에이전트 런타임`입니다.
