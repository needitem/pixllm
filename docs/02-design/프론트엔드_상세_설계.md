# 프론트엔드 상세 설계

목적: 현재 PIXLLM 프론트엔드를 실제 renderer 코드 기준으로 설명한다.

## 1. 범위

현재 프론트엔드는 별도 web SPA가 아니라 Electron renderer다.

실제 구조:

```text
desktop/src/renderer/
  App.svelte
  app.css
  main.ts
  lib/
    api.ts
    bridge.ts
    store.ts
```

예전 문서의 `features/team`, `features/bridge` 같은 구조는 현재 구현이 아니라 미래 설계에 가깝다.

## 2. 현재 책임

- settings 로드/저장
- workspace 선택과 session 목록 표시
- session 생성/복구/저장
- local agent stream 이벤트 구독
- conversation과 status timeline 렌더링
- backend runs 목록 조회
- run detail의 approvals, tasks, artifacts 렌더링
- approve/reject 액션 트리거

## 3. 데이터 흐름

```mermaid
flowchart LR
    A["User Input"] --> B["Renderer App.svelte"]
    B --> C["bridge.ts / preload"]
    C --> D["Electron Main"]
    D --> E["queryEngineService / QueryEngine"]
    E --> C
    C --> B
    B --> F["Runs API via preload"]
    F --> G["backend /api/v1/runs"]
```

renderer는 크게 두 종류의 데이터를 받는다.

- local agent stream events
- backend run 조회 결과

## 4. 현재 상태 관리 특성

현재는 거대한 전역 store보다 `App.svelte` 중심 상태와 보조 helper 구조에 가깝다.

주요 상태 범주:

- settings
- conversation
- session metadata
- stream progress
- runs list
- selected run detail
- approvals / artifacts / tasks

## 5. 현재 강점과 한계

강점:

- 현재 동작 추적이 쉽다.
- preload bridge와 API helper가 분리돼 있다.
- local session 화면과 backend run inspector가 같이 연결돼 있다.

한계:

- `App.svelte` 책임이 아직 크다.
- session UI와 runs UI가 더 세분화될 여지가 있다.
- timeline row, artifact panel, approval panel의 재사용성이 낮다.

## 6. 현재 문서에서 제거할 전제

- 별도 web SPA
- team/bridge 상태 표시
- 이미 완성된 feature-folder 구조
- remote execution 전용 frontend lane
- MCP/open-world UI surface

현재 PIXLLM 프론트엔드는 `desktop renderer 단일 앱`으로 설명하는 것이 맞다.
