# 모니터링 상세 설계

목적: 현재 PIXLLM에서 실제로 수집 가능한 실행 추적과 부족한 지점을 정리한다.

## 1. 현재 관측 범위

현재 코드 기준 주요 관측 단위:

- `desktop_session`
- `local_turn`
- `tool_batch`
- `local_task`
- `terminal_capture`
- `backend_run`
- `backend_approval`

`bridge_session`, `team_worker`, `plugin/skill telemetry`는 현재 기본 관측 범위가 아니다.

## 2. 현재 데이터 소스

| 소스 | 위치 | 내용 |
|---|---|---|
| 세션 저장소 | `desktop/src/main/sessions.cjs` | 메시지, status events, local trace, run snapshot |
| local agent stream | `queryEngineService.cjs` -> `agent:stream-event` | request 시작, transition, tool use/result, token, done, error |
| task runtime | `desktop/src/main/tasks/taskRuntime.cjs` | background task 상태, terminal captures |
| backend runs API | `/api/v1/runs*` | run 상태, tasks, approvals, artifacts |
| backend health API | `/api/v1/health` | 서버 상태 확인 |

## 3. renderer가 직접 보는 신호

stream events:

- `request_start`
- `session_restored`
- `status`
- `model`
- `assistant_message`
- `tool_use`
- `tool_result`
- `transition`
- `tool_batch_start`
- `tool_batch_end`
- `terminal`
- `user_question`
- `brief`
- `token`
- `done`
- `cancelled`
- `error`

session에 저장되는 값:

- 메시지 기록
- `statusEvents`
- `localTrace`
- `localSummary`
- `localError`
- `runSnapshot`

## 4. 현재 강한 부분

- local session trace가 구조화돼 있다.
- 어떤 tool이 어떤 입력으로 호출됐는지 복원 가능하다.
- background shell/build는 task runtime과 terminal capture로 분리 저장된다.
- backend runs가 있으면 approvals와 artifacts를 UI에서 같이 볼 수 있다.

## 5. 현재 약한 부분

- 중앙 telemetry backend 없음
- local turn과 backend run을 강하게 잇는 공통 correlation id 부족
- denial/failure/latency를 장기 집계하는 metrics store 없음
- 대시보드형 분석은 아직 없음

즉 현재는 `실행 추적은 충분하지만 중앙 관측 플랫폼은 약하다`가 정확한 설명이다.

## 6. 운영에서 먼저 볼 것

1. `agent:stream-event`가 정상적으로 흐르는지
2. session의 `localTrace`와 `statusEvents`가 유지되는지
3. `taskRuntime`에 실패 task와 terminal capture가 남는지
4. backend `/api/v1/runs`에서 approvals/artifacts 상태가 맞는지
5. `/api/v1/health`와 LLM endpoint가 정상인지

## 7. 문서상 표현 원칙

- "구조화된 local trace가 있다"
- "runs/approvals/artifacts 조회 표면이 있다"
- "중앙 telemetry stack은 아직 없다"
- "bridge/team/plugin/skill 관측은 현재 범위 밖이다"

현재 PIXLLM 모니터링은 `local session 추적 + backend run 조회`를 중심으로 설명하는 게 맞다.
