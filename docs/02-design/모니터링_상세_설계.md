# 모니터링 상세 설계

> 목적: PIXLLM 관측 대상을 `백엔드 헬스 체크`에서 `세션/턴/도구/태스크/브리지` 중심으로 확장

## 1. 관측 단위

새 구조에서 모니터링의 기본 단위는 아래입니다.

- `session`
- `turn`
- `tool_call`
- `task`
- `approval`
- `bridge_session`
- `model_usage`

## 2. 핵심 지표

| 범주 | 지표 예시 |
|---|---|
| Session | 생성 수, 재개율, 평균 길이, 중단 이유 |
| Turn | 첫 토큰 시간, 완료 시간, 실패율 |
| Tool | 호출 수, 성공률, 평균 duration, permission denial 비율 |
| Task | 생성 수, retry 수, cancel 수, terminal state 분포 |
| Approval | 요청 수, 승인 대기 시간, 거절 비율 |
| Bridge | active remote sessions, poll latency, reconnect count |
| Model | token usage, turn cost, completion latency |
| Plugin/Skill | 활성화 수, 실패한 로드 수, 도구 사용량 |

## 3. 로그 구조

모든 실행 로그는 아래 correlation key를 공유해야 합니다.

- `session_id`
- `turn_id`
- `task_id`
- `agent_id`
- `tool_call_id`
- `approval_id`

로그 수준:

- `info`: 정상 상태 전이
- `warn`: 재시도, 약한 evidence, 느린 도구
- `error`: 실패한 task, bridge 중단, 모델 호출 실패

## 4. 대시보드 우선순위

### 운영 대시보드

- 활성 세션 수
- 실행 중 task 수
- bridge 연결 수
- approval backlog

### 품질 대시보드

- tool failure hotspot
- verification failure rate
- retry-heavy task 유형
- artifact 없는 final response 비율

### 비용 대시보드

- 모델별 토큰 사용량
- 세션당 평균 비용
- 도구 사용 대비 모델 사용 비율

## 5. 경보 기준

- bridge poll 실패가 연속 임계치를 넘을 때
- task 실패율이 비정상적으로 높을 때
- approval backlog가 장시간 해소되지 않을 때
- 모델 응답 지연이 임계치를 넘을 때
- plugin/skill 로딩 실패가 급증할 때

## 6. 유지되는 기존 축

모델과 데이터 평면에 대해서는 기존 헬스 체크가 계속 필요합니다.

- `vLLM` 모델 목록 확인
- embeddings/vector store 상태
- Redis/MinIO/Qdrant 같은 지원 서비스 상태

다만 이것은 전체 모니터링의 일부일 뿐, 더 이상 전부가 아닙니다.

## 7. 권장 수집 방식

- 로컬 앱은 구조화 로그와 session trace를 남깁니다.
- control plane은 task/event/approval/bridge 상태를 저장합니다.
- 모델 평면은 usage와 latency만 요약 수집합니다.

PIXLLM 모니터링은 이제 `서비스가 살아 있는가`를 넘어서 `어떤 세션이 어디서 막히는가`를 보여줘야 합니다.
