# 모니터링 상세 설계

> 목적: 현재 PIXLLM에서 실제로 수집 가능한 실행 흔적과 부족한 지점을 정리

## 1. 현재 관측 단위

현재 코드에서 실질적으로 관측 가능한 단위는 아래입니다.

- `desktop_session`
- `local_turn`
- `tool_batch`
- `local_task`
- `terminal_capture`
- `backend_run`
- `backend_approval`

과거 문서에 있던 `bridge_session`, `team_worker`, `plugin/skill telemetry`는 현재 로컬 경로의 관측 단위가 아닙니다.

## 2. 현재 데이터 소스

| 소스 | 저장/전송 위치 | 내용 |
|---|---|---|
| 세션 저장소 | `desktop/src/main/sessions.cjs` | 메시지, status events, local trace, run snapshot |
| 로컬 에이전트 스트림 | `local_agent_service.cjs` -> `agent:stream-event` | 요청 시작, 상태 전이, tool use/result, token, done, error |
| 로컬 task runtime | `local_task_runtime.cjs` | background task 상태, terminal captures |
| backend runs API | `/api/v1/runs*` | run 상태, task 목록, approvals, artifacts |
| backend health API | `/api/v1/health` | 서버 헬스 확인 |

## 3. 현재 수집되는 주요 신호

### 로컬 스트림 이벤트

현재 데스크톱이 직접 받는 이벤트는 아래입니다.

- `request_start`
- `session_restored`
- `status`
- `model`
- `assistant_message`
- `tool_use`
- `tool_result`
- `transition`
- `tool_batch_start`
- `tool_batch_end`
- `terminal`
- `user_question`
- `brief`
- `token`
- `done`
- `cancelled`
- `error`

### 세션 저장 필드

현재 세션 파일에는 아래 종류의 데이터가 들어갑니다.

- 대화 메시지
- `statusEvents`
- `localTrace`
- `localSummary`
- `localError`
- `runSnapshot`

### task / terminal 상태

현재 로컬 task runtime이 보관하는 값:

- task summary
- background 여부
- PID
- 종료 코드
- 최근 output
- terminal capture 이력

## 4. 현재 모니터링에서 강한 부분

- 로컬 대화 단위 trace가 남습니다.
- 어떤 tool이 어떤 입력으로 호출됐는지 session trace에서 복원할 수 있습니다.
- background shell/build task 결과를 별도 runtime 문서와 terminal capture로 남깁니다.
- backend runs가 있으면 approvals와 artifacts를 UI에서 재조회할 수 있습니다.

## 5. 현재 모니터링에서 약한 부분

현재는 아직 아래가 약합니다.

- 중앙 집계형 telemetry backend
- 모든 로컬 turn과 backend run을 잇는 공통 correlation id
- latency / error / denial 비율을 장기 보관하는 metrics store
- 대시보드형 시계열 분석

즉 지금은 "실행 흔적은 남지만, 지표 시스템은 얕다"가 더 정확한 설명입니다.

## 6. 운영 관점에서 먼저 봐야 할 것

현재 기준으로 운영자가 확인해야 할 우선순위는 아래입니다.

1. `agent:stream-event`가 정상적으로 흘렀는가
2. session 파일에 `localTrace`와 `statusEvents`가 저장됐는가
3. `local_task_runtime`에 실패한 task나 terminal capture가 남았는가
4. backend `/api/v1/runs`에서 approval/artifact 상태가 맞는가
5. `/api/v1/health`와 LLM endpoint가 정상인가

## 7. 권장 문서화 기준

현재 구현 기준으로 문서가 주장해야 하는 범위는 아래입니다.

- "구조화된 로컬 trace는 있다"
- "runs/approvals/artifacts 조회 표면은 있다"
- "중앙 telemetry stack은 아직 약하다"
- "bridge/team/plugin/skill 관측은 현재 범위 밖이다"

## 8. 다음 확장 우선순위

현재 코드와 가장 잘 맞는 다음 단계는 아래입니다.

1. local turn, tool denial, task failure에 대한 공통 event schema 고정
2. session trace와 backend run snapshot 사이의 연결 키 정리
3. renderer에서 terminal/tool failure hotspot을 더 쉽게 보이게 개선
4. 필요하면 그 다음에만 중앙 metrics store 도입

현재 PIXLLM 모니터링은 `분산 control plane`보다 `로컬 세션 추적 + backend run 조회`를 중심으로 설명하는 것이 맞습니다.
