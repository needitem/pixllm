# 인증보안 상세 설계

> 목적: PIXLLM 보안을 `API 토큰` 수준에서 `도구 권한 + 승인 + 플러그인 신뢰 + 원격 세션 보호` 수준으로 확장

## 1. 신뢰 경계

새 구조에서 주요 trust boundary는 아래입니다.

- 사용자 워크스페이스
- 데스크톱 런타임
- plugin/skill/MCP 확장
- control plane
- model/data plane
- remote worker

## 2. 기본 보안 원칙

- 파일 읽기와 쓰기 권한을 구분합니다.
- 위험 명령은 승인 없이는 실행하지 않습니다.
- 원격 세션 토큰은 세션 단위로 발급하고 만료시킵니다.
- plugin과 MCP는 신뢰 수준과 설치 출처를 기록합니다.
- 로그와 artifact에는 비밀값을 남기지 않습니다.

## 3. 권한 모드

| 모드 | 허용 범위 |
|---|---|
| `read_only` | 검색, 읽기, 요약 |
| `workspace_edit` | 승인된 파일 수정 |
| `command_execution` | 승인된 명령 실행 |
| `remote_execution` | 브리지 뒤 원격 세션 실행 |
| `admin` | 정책/설정/브리지 관리 |

## 4. Approval Gate

아래 작업은 approval을 요구합니다.

- 파일 편집과 쓰기
- 테스트/빌드/스크립트 실행
- 네트워크 접근
- 원격 세션 생성
- plugin 설치 또는 활성화

approval payload에는 최소한 아래가 있어야 합니다.

- 요약
- 대상 경로 또는 대상 환경
- 명령 미리보기
- 위험 수준
- 실행 후 남을 artifact 종류

## 5. Plugin and Skill Trust

plugin과 skill은 실행 확장점이므로 별도 관리가 필요합니다.

- 출처 식별
- 버전 기록
- 허용된 도구 범위 제한
- install/enable/disable 이력 저장

기본 원칙:

- 신뢰되지 않은 plugin은 읽기 전용으로 시작합니다.
- 쓰기/실행이 가능한 plugin은 명시적 승인 또는 정책 허용이 필요합니다.

## 6. Bridge and Remote Security

원격 세션은 특히 아래를 보장해야 합니다.

- environment registration 인증
- work lease와 heartbeat
- reconnect 시 세션 소유권 확인
- 세션 종료 후 worktree 정리
- remote artifact 전송 시 민감값 마스킹

## 7. 모델 평면과의 관계

모델 평면은 기존 방식을 유지합니다.

- 기존 API 토큰/게이트웨이 체계는 유지합니다.
- 바뀌는 것은 모델 호출 전후의 실행 보안 계층입니다.

## 8. 감사와 추적

아래 이벤트는 반드시 감사 가능해야 합니다.

- 승인 요청과 승인 결과
- 파일 수정과 diff 생성
- 명령 실행과 종료 코드
- remote session 생성과 중단
- plugin 활성화와 비활성화

PIXLLM 보안은 이제 "API 키를 아는가"보다 "누가 어떤 실행을 어떤 범위에서 승인받아 수행했는가"를 중심으로 설계해야 합니다.
