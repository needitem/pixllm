# 인증보안 상세 설계

목적: 현재 PIXLLM desktop과 backend 경로에 실제로 들어가 있는 보안 제어를 정리한다.

## 1. 현재 trust boundary

현재 주요 경계:

- 사용자 workspace
- Electron renderer
- Electron preload / main
- backend API
- LLM endpoint

plugin/skill/MCP 확장, remote worker, bridge session은 현재 기본 보안 경계가 아니다.

## 2. 현재 구현된 제어

### 2.1 Renderer -> preload / main

- preload가 허용된 desktop surface만 노출한다.
- dev server URL은 `localhost` 또는 `127.0.0.1`만 허용한다.
- main process는 임의 외부 origin 로드를 차단한다.

### 2.2 settings와 token

- `settings.cjs`가 설정을 관리한다.
- API token과 LLM token은 Electron `safeStorage` 가능 시 보호 저장한다.
- 복호화는 main process에서만 수행한다.

### 2.3 workspace confinement

`workspace.cjs`와 tool validation이 아래를 강제한다.

- workspace path 정규화
- realpath 기준 workspace 내부 경로 확인
- path traversal 차단
- workspace-relative path 강제

### 2.4 local tool validation / policy

현재 모든 tool call 앞에서 아래가 걸린다.

- input schema validation
- workspace-relative path validation
- active tool scope 확인
- grounded path 확인
- read-before-edit / read-before-overwrite
- execution intent 또는 충분한 context 확인
- backend reference path와 local file path의 역할 분리

이 제어는 현재 `Tool.cjs`, `ToolRuntime.cjs`, `hooks/useCanUseTool.cjs`, `tools/*`에 나뉘어 있다.

현재 Qwen adapter가 malformed tool-call을 복구하더라도, 복구된 tool call은 그대로 실행되지 않는다. 반드시 같은 policy 계층을 다시 통과해야 한다.

### 2.5 shell 안전장치

`workspace.cjs`의 shell/build 실행에는 아래가 있다.

- 위험 명령 패턴 차단
- allowlist/prefix 제어
- timeout
- workspace cwd 고정

## 3. approval 표면

현재 approval은 두 층이다.

- local tool policy: 즉시 허용/거절
- backend runs approval: `/api/v1/runs/{run_id}/approvals/*`

즉 모든 local tool call이 backend approval inbox를 거치는 구조는 아니다.

## 4. Qwen 적응층과 보안

현재 모델 경계에서 일어나는 보안상 중요한 사실:

- Qwen textual `<tool_call>`은 parser가 받아도 신뢰 입력이 아니다.
- malformed JSON-like payload를 복구해도, tool 이름과 인자는 다시 schema/policy 검사를 거친다.
- 한국어 rewrite hint는 search/symbol 힌트 보조용이고, 곧바로 파일 수정 권한으로 연결되지 않는다.
- `company_reference_search` 결과는 grounded evidence로는 인정되지만 local edit 대상은 아니다.

## 5. 현재 보호하는 자산

| 자산 | 현재 보호 방식 |
|---|---|
| workspace 파일 | confinement, path gate, read-before-edit |
| 로컬 명령 실행 | allowlist, dangerous pattern 차단, timeout |
| API/LLM token | `safeStorage` 기반 보호 저장 |
| renderer bridge | localhost/file 제한 |
| backend approval action | explicit approve/reject API |

## 6. 현재 없는 것

- 범용 central approval broker
- plugin trust model
- MCP trust / install policy
- remote bridge token / lease / reconnect security
- team worker ownership enforcement

따라서 현재 보안 문서는 `확장 실행면 보안`보다 `workspace confinement + local tool gate + shell safety + Qwen parser 이후 재검증`을 중심으로 써야 맞다.
