# 인증보안 상세 설계

> 목적: 현재 PIXLLM 데스크톱과 backend 경로에 실제로 들어가 있는 보안 통제를 정리

## 1. 현재 신뢰 경계

현재 코드 기준 주요 trust boundary는 아래입니다.

- 사용자 워크스페이스
- Electron renderer
- Electron preload / main
- backend API
- LLM endpoint

과거 문서에 있던 `plugin/skill/MCP 확장`, `remote worker`, `bridge session`은 현재 로컬 에이전트 보안 경계의 기본 요소가 아닙니다.

## 2. 현재 구현된 보안 통제

### 2.1 Renderer -> preload / main 경계

- preload는 허용된 hostname만 노출합니다.
- dev server URL은 `localhost` 또는 `127.0.0.1`만 허용합니다.
- main 프로세스도 localhost 외 개발 서버를 거부합니다.

즉 renderer가 임의 원격 origin으로 바로 올라가는 구조가 아닙니다.

### 2.2 설정과 토큰 저장

- settings는 `desktop/src/main/settings.cjs`에서 관리합니다.
- API token과 LLM token은 Electron `safeStorage`가 가능하면 암호화 저장합니다.
- 복호화는 main 프로세스에서만 일어납니다.

### 2.3 워크스페이스 경계

`workspace.cjs`와 `local_tool.cjs`는 아래를 강제합니다.

- workspace path 정규화
- realpath 기준 workspace 내부 경로 확인
- path traversal 차단
- workspace-relative path 요구

즉 로컬 파일 도구는 기본적으로 워크스페이스 밖 경로를 바로 받지 않습니다.

### 2.4 로컬 tool validation / policy

현재 로컬 tool 호출 전에는 아래 검사가 걸립니다.

- schema 기반 입력 검증
- 허용된 인자 타입과 필수값 검사
- grounded path 여부 검사
- read-before-edit 검사
- execution intent 여부 검사

이 검사는 `local_tool.cjs`, `local_tools.cjs`, `local_tool_policy.cjs`에 나뉘어 있습니다.

### 2.5 셸 실행 안전장치

`workspace.cjs`의 shell 경로는 아래를 가집니다.

- 위험 명령 패턴 차단
- 허용된 command prefix allowlist
- timeout
- workspace cwd 고정

예를 들어 destructive delete, `git reset --hard`, `svn revert` 같은 패턴은 막힙니다.

## 3. 현재 approval 의미

현재 approval은 두 층으로 나뉩니다.

- 로컬 agent tool policy: 즉시 허용/거절
- backend runs approval: `/api/v1/runs/{run_id}/approvals/*`

즉 모든 로컬 tool call이 backend approval inbox를 거치는 구조는 아닙니다.

현재 backend approval은 주로 run detail 조회와 승인/거절 API로 존재합니다.

## 4. 현재 보호되는 자산

| 자산 | 현재 보호 방식 |
|---|---|
| 워크스페이스 파일 | workspace 내부 경로 강제, read-before-edit, tool policy |
| 로컬 명령 실행 | allowlist, dangerous pattern 차단, timeout |
| API/LLM token | `safeStorage` 기반 암호화 저장 가능 |
| renderer bridge | localhost/file 기준 허용 |
| backend approval action | 명시적 approve/reject API |

## 5. 현재 없는 것

현재 코드에 없는 보안 계층은 아래입니다.

- 범용 central approval broker
- plugin trust model
- MCP trust / install policy
- remote bridge token / lease / reconnect security
- team worker ownership enforcement

즉 과거 문서의 "확장 실행면 보안"은 현재 구현 사실이 아니라 미래 설계 조각이었습니다.

## 6. 현재 기준의 보안 설명 원칙

현재 PIXLLM 보안은 아래처럼 설명하는 것이 맞습니다.

- Electron 경계와 localhost 제약이 있다
- settings token 저장 보호가 있다
- workspace path confinement가 있다
- local tool schema + policy gate가 있다
- shell allowlist와 dangerous command 차단이 있다
- backend run approval surface는 있지만 모든 로컬 실행의 중앙 승인 시스템은 아니다

## 7. 다음 우선순위

현재 코드 기준으로 가장 현실적인 다음 보안 보강은 아래입니다.

1. local tool denial / shell rejection audit event를 더 구조화
2. backend run approval과 local tool policy의 역할 구분을 문서와 UI에 명확화
3. write / shell / build에 대한 더 세밀한 정책 모드 도입
4. 그 다음에야 필요 여부를 보고 remote/team 보안 모델을 설계

현재 PIXLLM의 보안 문서는 `확장점 신뢰 모델`보다 `로컬 워크스페이스 경계와 tool gate`를 중심으로 쓰는 것이 정확합니다.
